Fixing License Check Issues For Project Content

Aug 12, 2025 by ADMIN 48 views

Hey everyone, I've noticed some issues popping up in IPLab, specifically related to the license check action. I believe these issues are primarily affecting project content. Let's dive in and figure out what's going on and how we can fix it.

Identifying the Problem: License Check and Project Content

So, I've been looking at a few specific issues that seem to be related to the license check, and they all point towards project content. I'm talking about things like crates hosted in our repositories. Here are the issues I'm referring to:

  • crate/cratesio/-/rust_test_scenarios/0.1.0 (https://gitlab.eclipse.org/eclipsefdn/emo-team/iplab/-/issues/22575)
  • crate/cratesio/-/logging_tracing/0.1.0 (https://gitlab.eclipse.org/eclipsefdn/emo-team/iplab/-/issues/21880)
  • crate/cratesio/-/async_runtime/0.1.0 (https://gitlab.eclipse.org/eclipsefdn/emo-team/iplab/-/issues/21889)
  • crate/cratesio/-/rust-kvs/0.1.0 (https://gitlab.eclipse.org/eclipsefdn/emo-team/iplab/-/issues/21843)

Am I right in thinking that these issues are all about project content? My initial assessment is that they are. If you're nodding along, great! If not, let me know, and we can investigate further. Understanding the scope of the problem is the first step to finding a solution. Recognizing the source of the problem helps to eliminate unnecessary steps.

The core of the issue is that the Eclipse Dash License Tool, which is used for these checks, isn't really designed to handle project content in the same way it handles third-party content. The tool works best when dealing with content from well-defined software repositories, like crates.io, which is the source of the packages being flagged. This mismatch creates these issues.

If we were to simply close these issues, they'd likely just reappear the next time the license check action runs. That's not a sustainable solution, is it? We need a more permanent fix to prevent these issues from constantly resurfacing. The goal is to ensure that the license check action runs smoothly without being bogged down by false positives related to project content.

Proposed Solution: Excluding Project Content from License Checks

The most straightforward solution, in my opinion, is to exclude these project content items from the license check entirely. This would prevent the tool from even looking at them, thus eliminating the generation of these issues. It is a matter of configuring the checks to ignore these specific items, similar to how we might ignore certain files or directories in other automated processes.

Here's a rough example of how we could potentially achieve this using a bash script. Keep in mind, this is just a starting point, and we can certainly optimize it:

$ cargo tree -e normal --prefix none --no-dedupe \
| sort -u \
| grep -v '^[[:space:]]*{{content}}#39; \
| grep -v rust_test_scenarios \
| grep -v logging_tracing \
| grep -v async_runtime \
| grep -v rust-kvs \
| sed -E 's|([^ ]+) v([^ ]+).*|crate/cratesio/-/
| java -jar org.eclipse.dash.licenses-<version>.jar -
