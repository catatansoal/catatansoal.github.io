Fixing Technical Issues: JOSS Review For Tissue Data Explorer

Aug 8, 2025 by ADMIN 62 views

Hey everyone! Taylor here, diving into the technical review for this project to get it ready for publication on JOSS. This is the first of two discussions, and we're focusing on the nitty-gritty technical aspects that need some love before we can give it the green light.

Code Issues

Let's start by talking about the code. We've got a few things to address here, but nothing too scary. Think of it as polishing the final product to make it shine!

Security

Security is super important, and one small tweak can make a big difference. Currently, the Dockerfile in app/Dockerfile is set up to run as the root user. This isn't ideal from a security standpoint. We want to minimize the potential damage if someone manages to break into the container. Running as a non-root user adds an extra layer of protection. Think of it as having a bodyguard for your application!

To fix this, we need to modify the Dockerfile to create a non-root user and run the application under that user. It might sound complicated, but it's actually a pretty common practice. There are tons of great resources out there to help you, like this helpful guide. It walks you through the process step-by-step. It might seem like a small detail, but it's a crucial one for ensuring the long-term security of the application and its users. By implementing this change, we're not just patching a hole; we're building a more secure foundation for the project. This is not a big problem but would be great to be addressed.

Code Fixes

Next up, we have a little syntax hiccup in the frontend code. In config_portal/config_components/ui.py, line 149, we need to make a small adjustment to align with React syntax. Specifically, we need to change margin-top to marginTop. You know, the little things can sometimes trip us up, right?

This change will clear up an error that's currently showing up in the Frontend console logs. It's like getting rid of that annoying flickering lightbulb – a simple fix that makes a big difference in the overall experience. This might seem like a minor change, but it's important for maintaining a clean and error-free console, which can be invaluable for debugging and future development. It's these small details that contribute to a polished and professional application. So, let's get that lightbulb fixed!

README's Docker Cleanup Command

Now, let's talk about keeping things tidy. The README.md file includes a cleanup section with instructions for removing Docker resources. Currently, it suggests using docker image prune -a. While this command does the job, it's a bit like using a sledgehammer to crack a nut. It removes all unused images, which could inadvertently affect other Docker-based solutions running on the same system. It’s a bit like cleaning your room by throwing everything out the window – effective, but maybe not the most considerate approach.

A better approach would be to target only the resources associated with this specific application. It’s about being a good neighbor in the Docker ecosystem.

Here's a suggestion for a more targeted cleanup process:

docker rm -f $(docker ps -aq --filter "name=tissue-data-explorer")
docker rmi -f $(docker images -q --filter=reference='tissue-data-explorer*')
docker volume rm config-data-dev
docker builder prune

This set of commands is longer, yes, but it's much safer. It specifically removes containers, images, and volumes related to the tissue-data-explorer application. It's like using a scalpel instead of a chainsaw – more precise and less likely to cause collateral damage. This is crucial for users who might be running other Dockerized applications on their systems. By implementing this change, we're not just cleaning up after ourselves; we're also ensuring that we don't accidentally mess up someone else's setup. It's a bit of extra effort that shows we care about the user experience and the wider Docker community.

Some Suggestions

Now, let's brainstorm some improvements that aren't strictly necessary for publishing, but would definitely enhance the user experience. Think of these as the cherry on top of an already delicious sundae!

Image Viewer

When viewing an image (e.g., S1-1-1), the slider currently only updates the image when the cursor is released. It's like waiting for a loading bar to finish before you see the result – a bit clunky.

A smoother experience would be to update the image on each tick of the slider. This would provide instant feedback and make it easier for users to quickly browse through the images. It would be like having a real-time preview, making the image exploration process much more intuitive and engaging. It's a small change that could make a big difference in how users interact with the images.

Navigation

Navigation is key to a user-friendly application. If users can't easily find their way around, they're likely to get frustrated and give up. Let's explore some ways to make navigation even smoother.

  • Breadcrumbs on all sub-pages: Currently, when you're deep within a section (e.g., viewing volumetric data), it can be tricky to return to the home page. Breadcrumbs – those little navigation trails at the top of the page – would make it super easy to jump back to previous sections. It's like leaving a trail of breadcrumbs (hence the name!) so users can easily retrace their steps. This is especially helpful for complex applications with multiple levels of navigation.
  • Clickable application logo: Another simple but effective navigation enhancement would be to make the application logo clickable, redirecting users to the home page. This is a common convention that users are familiar with, and it provides a quick and intuitive way to get back to the main screen. It's like having a home button that's always within reach.
