Bug Report: OpenAI gpt-oss Fails in NeMo Agent Toolkit

Hey everyone, I've run into a snag while using the NeMo Agent Toolkit, specifically when trying to get it to play nice with the OpenAI gpt-oss:20b model. I've been experimenting with the email phishing example, and while things work smoothly with the gemma3:12b model via the Ollama provider, switching over to gpt-oss:20b leads to errors. I've dug into the details, and it seems like there might be a bug related to how the Agent Toolkit interacts with this particular model.

The Setup: NeMo Agent Toolkit, Ollama, and OpenAI

I've configured the Ollama provider to be able to use a variety of models. I'm running the email phishing example, which is a great way to test the toolkit's capabilities. The example is designed to simulate an agent that can analyze and respond to phishing emails. My initial tests with the gemma3:12b model worked flawlessly. This gave me confidence that my setup was correct and the toolkit was functioning as expected. After this success, I decided to expand my testing by trying out the gpt-oss:20b model. This model is known for its advanced capabilities, and I was eager to see how it would perform within the NeMo Agent Toolkit framework.

However, things took a turn for the worse when I switched to gpt-oss:20b. The agent started throwing errors, and the email phishing example failed to execute correctly. This led me to suspect a potential incompatibility or a bug related to the interaction between the Agent Toolkit and the gpt-oss:20b model. The errors I encountered pointed towards issues with parsing the agent's output, as the toolkit was unable to correctly interpret the model's responses. This problem prevented the agent from performing its tasks, essentially breaking the example. It's important to note that the core setup, including the Ollama provider and the NeMo Agent Toolkit, was the same across both tests. The only change was the AI model used. Given that the toolkit worked with gemma3:12b and failed with gpt-oss:20b, it strengthens the hypothesis that the problem lies within the interaction with this specific model. This failure highlighted the importance of ensuring compatibility between the NeMo Agent Toolkit and various AI models, particularly those offered by providers like OpenAI.

Bug Details and Error Logs

The core issue seems to be a parsing problem within the NeMo Agent Toolkit when it uses gpt-oss:20b. The toolkit's attempts to parse the model's output repeatedly failed, as evident in the error logs. Specifically, the agent was unable to correctly interpret the responses from the model, leading to repeated retries and ultimately, failure. The error messages indicated an issue with the format of the output, specifically the absence of the expected Action: after Thought:. This suggests that the model's responses did not adhere to the format the toolkit expected, causing the parsing to fail. The logs show the agent retrying to parse the output multiple times, highlighting the persistence of the issue. After three attempts to parse the output, the agent gave up, further confirming the presence of a significant parsing problem. This series of retries underscores the challenges in ensuring seamless interaction between the toolkit and various AI models, especially when it comes to output formatting and parsing. The root cause is likely related to the difference in the expected output format by the NeMo Agent Toolkit and the actual format generated by the OpenAI gpt-oss:20b model. This mismatch caused the parsing mechanism to fail repeatedly, resulting in the observed errors.

I set up the environment to use the gpt-oss:20b model through the Ollama provider. I ran the email phishing example, which requires no special code modifications as it is designed to be run directly. The expected behavior was for the agent to analyze the email and provide a suitable response. Instead, I encountered an error related to parsing the agent's output. The error message stated that the expected format was not met; in particular, the Action: tag was missing after the Thought: tag. This is a critical breakdown, as the toolkit depends on properly formatted output to execute actions and continue the workflow. This failure suggests a need for the toolkit to either be updated to better parse the output from gpt-oss:20b or for the model's output to be adjusted to match the expected format. Addressing this parsing error is crucial to ensuring the reliability and functionality of the NeMo Agent Toolkit when using the gpt-oss:20b model, as this bug severely impacts its ability to perform its intended tasks.

Steps to Reproduce

To reproduce this bug, you don't need to write any special code. The issue arises when you run the example within the NeMo Agent Toolkit and select gpt-oss:20b as the AI model. The toolkit uses the model to analyze and respond to phishing emails. The expectation is for the agent to analyze the provided email and generate a reasoned response. The bug manifests in the form of repeated parsing failures, leading to the overall failure of the agent's operations. This occurs because the agent's output is not correctly parsed by the toolkit. Therefore, the expected behavior is not achieved, and the example fails to complete successfully. When running the email phishing example with gpt-oss:20b, the NeMo Agent Toolkit attempts to parse the agent's output, which is the response generated by the AI model. However, the agent's output is formatted in a way that is not compatible with the toolkit's parsing logic. This incompatibility causes the parsing process to fail, resulting in the Invalid Format error.

To recreate the issue, one needs to:

  1. Configure the Ollama provider to use the gpt-oss:20b model.
  2. Run the email phishing example.
  3. Observe the error messages related to output parsing.
  4. Confirm the agent fails to execute the task correctly.

Potential Solutions and Workarounds

To tackle this bug, a few solutions could be considered. First, updating the NeMo Agent Toolkit to be able to handle the output format from gpt-oss:20b might be the most direct approach. This could involve adjusting the parsing mechanisms within the toolkit to correctly interpret the responses from the model. Second, modifying the model's output format could be another way to solve the problem. This would require either retraining the model or tweaking its configuration to generate outputs that align with the toolkit's expectations. Third, users could consider increasing the parse_agent_response_max_retries setting. This change would allow the toolkit to attempt parsing the output for a greater number of times before giving up. As a workaround, users could try other AI models compatible with the NeMo Agent Toolkit. The gemma3:12b model, for example, has been proven to work. In the meantime, users could potentially employ a preprocessing step that reformats the model's output to align with the toolkit's requirements before it's parsed. This pre-processing could include a script or a custom function that converts the output into the correct format. To help with this problem, providing clear documentation on output formatting requirements for different models would also be really helpful. This documentation would assist users in setting up and troubleshooting their agents and making sure that the AI models' responses are correctly parsed. Moreover, incorporating more comprehensive error messages with guidance on potential causes and resolutions would make debugging much easier.
