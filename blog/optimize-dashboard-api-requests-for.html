Understanding the Dashboard API Optimization Problem

To effectively tackle the performance and stability issues with our dashboard API requests, we first need to deeply understand the core problems. Currently, the simultaneous nature of multiple API requests when loading dashboards is overwhelming our backend servers. This issue doesn't just slow down the loading times; it can lead to server timeouts and crashes, creating a frustrating experience for our users and operational headaches for us. The essence of the problem lies in the sheer volume of requests and the lack of coordination between them.

Each dashboard, whether it's for a Property Manager, Landlord, Tenant, or Admin, is designed to display a variety of information, from summary statistics to detailed tables and notifications. To populate these dashboards, the frontend makes several independent API calls. The PMDashboardPage, LandlordDashboardPage, TenantDashboardPage, and AdminDashboardPage, all trigger multiple API calls on load. These calls aren't happening sequentially; instead, many of them fire off in parallel to speed up the loading process. However, this parallel execution creates a bottleneck on the backend, as it struggles to handle numerous requests at the same time. It‚Äôs like trying to push a dozen cars through a one-lane tunnel ‚Äì congestion is inevitable.

Adding to the complexity, our context providers‚ÄîAuthContext, PropertyContext, LeaseContext, and NotificationContext‚Äîare also making their own API calls upon initialization. These contexts are designed to manage and provide application-wide data, such as user authentication status, property details, lease information, and notifications. However, when each context provider independently makes API calls, it exacerbates the issue of too many concurrent requests. It‚Äôs not just the dashboards themselves but also the underlying data providers contributing to the overload. Furthermore, many of these API calls are redundant, meaning they fetch the same or similar data multiple times. This redundancy not only wastes server resources but also increases the overall load time for the dashboards. In essence, we are fetching the same information multiple times when we could be doing it more efficiently. The lack of coordination between API calls and the presence of redundant requests are key factors contributing to the performance issues we are seeing. Therefore, our optimization strategy must address both the number of requests and the efficiency with which we fetch data. By reducing the number of API calls, eliminating redundancy, and improving backend handling, we can significantly enhance the performance and stability of our dashboards.

Implementing API Request Batching and Response Caching

To really get our dashboards running smoothly, we're focusing on two major strategies: API request batching and response caching. These methods are key to reducing the load on our backend and speeding up the user experience. Let's dive into how these solutions work and how we plan to implement them.

API request batching is like grouping a bunch of small packages into one big shipment. Instead of sending many individual API requests, we combine related requests into a single, more efficient call. This dramatically reduces the overhead on the backend and the number of round trips between the client and server. To achieve this, we're going to create a new batching service. This service will act as a central hub, collecting and combining multiple data requests into a single, optimized request. Think of it as a smart traffic controller that merges lanes to ease congestion. Along with the batching service, we‚Äôll add a new endpoint on the backend designed to handle these combined data requests. This endpoint will be capable of processing multiple requests simultaneously, streamlining the data retrieval process. It‚Äôs like building a multi-lane highway to handle the increased traffic flow. This approach not only reduces the number of requests but also allows the backend to optimize data fetching, potentially reducing database queries and processing time. By batching requests, we can cut down on network latency and server load, resulting in faster dashboard loading times.

Response caching takes a different but equally effective approach. It's like having a good memory‚Äîwe store recently fetched data on the client side and reuse it when needed, instead of repeatedly asking the server for the same information. This not only reduces the load on the backend but also provides a snappier user experience, as data is readily available. We plan to implement client-side caching for dashboard data, complete with appropriate Time To Live (TTL) settings. The TTL determines how long the cached data remains valid, ensuring we don‚Äôt serve stale information. We can use React Query, a powerful library that simplifies data fetching and caching in React applications, or we can create a custom caching layer. Either way, the goal is to store recently fetched data and serve it from the cache whenever possible. It‚Äôs like having a personal data bank that quickly provides the information you need. Proper cache invalidation is crucial to ensure that the cached data remains up-to-date. When data changes, we need to refresh the cache to reflect the latest information. This involves setting up mechanisms to detect data changes and trigger cache updates. Caching not only reduces server load but also improves perceived performance for users, as data loads almost instantly from the cache. By combining API request batching and response caching, we create a robust system that minimizes backend load, reduces network traffic, and delivers a faster, more responsive dashboard experience. These strategies work together to ensure that our dashboards are both performant and stable, providing a solid foundation for our application.
