The Issue: Synchronous getRedirectionURL Methods and TypeScript Errors

When working with SuperTokens in a TypeScript project, you might encounter a situation where you want to define your getRedirectionURL method synchronously. This is perfectly valid, especially when the logic for determining the redirection URL is simple and doesn't involve any asynchronous operations. However, the current type definition in SuperTokens expects getRedirectionURL to be an asynchronous function, returning a Promise. This mismatch between your synchronous implementation and the expected asynchronous type leads to TypeScript errors, as illustrated in the GitHub issue.

Specifically, the error arises because TypeScript enforces strict type checking. If you define getRedirectionURL as a synchronous function, TypeScript will complain that it doesn't match the expected Promise return type. This forces developers into a couple of less-than-ideal workarounds:

  1. Marking the method as async unnecessarily: To satisfy the TypeScript compiler, you might mark your synchronous getRedirectionURL method as async. While this resolves the type error, it introduces an unnecessary overhead. The async keyword implicitly wraps the return value in a Promise, even though no actual asynchronous operation is performed. This can slightly impact performance, although the effect is usually negligible.
  2. Ignoring the TypeScript error: Another workaround is to simply ignore the TypeScript error, perhaps by using a type assertion or disabling the specific linting rule. However, this approach undermines the benefits of using TypeScript in the first place. By ignoring type errors, you risk introducing runtime bugs and making your code harder to maintain.

Diving Deep: Understanding the Code and the Proposed Solution

Leveraging Promiseable for Cleaner Code
