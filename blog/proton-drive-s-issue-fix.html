Proton Drive S Issue: Fix Compatibility Problems

Aug 3, 2025 by ADMIN 49 views

Hey guys! Ever run into a weird issue where your Proton games try to install on a drive you didn't expect? This article dives deep into a compatibility report about a strange "Drive S" issue with Proton, specifically when using GE-Proton10-10. We'll break down the problem, explore potential causes, and provide a troubleshooting guide to get your games running smoothly. Let's get started!

Compatibility Report: The Elder Scrolls Online and the Mysterious Drive S

Our main focus is a compatibility issue encountered while trying to run The Elder Scrolls Online (ESO) through Steam using GE-Proton10-10. The user, mercster, reported that the game, along with other standalone .EXE installers run through Lutris, was attempting to install files on an unexpected "S:" drive. This is quite unusual, as Proton typically maps game installations to the "C:" drive within its virtualized Windows environment. This issue can be frustrating because it deviates from the standard behavior and can lead to confusion about where game files are actually being stored. Understanding the root cause of this issue is crucial for ensuring a seamless gaming experience with Proton. We'll explore the user's system setup, the symptoms they encountered, and potential steps to reproduce this behavior. By thoroughly examining the problem, we can develop effective troubleshooting strategies and prevent similar issues in the future. So, if you've ever wondered why your game is trying to install on a drive you didn't specify, stick around, because we're about to unravel this mystery!

System Information

To give you the full picture, here’s a rundown of the system configuration where this issue popped up:

  • GPU: 2080Super
  • Driver/LLVM version: 575.64.05
  • Kernel version: 6.15.8
  • Full system information report: https://gist.github.com/mercster/de19c728828d934cefcea3fdf54083d6
  • Proton version: GE-Proton10-10

This information is critical because it helps us understand the specific environment in which the problem occurred. Different hardware, drivers, and kernel versions can sometimes interact unexpectedly with Proton, leading to compatibility issues. For example, a particular driver version might have a bug that causes Proton to misinterpret drive mappings, or a specific kernel version might introduce changes that affect how Proton interacts with the file system. By providing this detailed system information, mercster has given us a valuable starting point for diagnosing the issue. It allows us to compare their setup with other users who might be experiencing similar problems and identify any common factors that could be contributing to the Drive S issue. Furthermore, this information is essential for developers who are working on Proton, as it helps them reproduce the problem in a controlled environment and develop a fix.

Symptoms: The Curious Case of the S: Drive

The main symptom is that when ESO is launched for the first time, the launcher prompts the user for an installation directory and defaults to "S:\common\Zenimax Studios" (or something similar). This is unexpected because Proton is designed to typically use the “C:” drive within its virtual environment for game installations. This issue isn't limited to ESO; the user also observed similar behavior when installing a game via Lutris using a standalone .EXE installer and GE-Proton10-10. In that instance, files were also being directed to the mysterious “S:” drive. The user discovered that “S:” was mapped to a location on their large data drive (a platter-based HDD mounted at /run/media/username/Data). This raises several questions about how Proton is handling drive mappings and why it’s deviating from the expected “C:” drive behavior. It’s crucial to understand what factors are influencing this drive mapping, as it could potentially affect other games and applications run through Proton. The user's workaround involved deleting the “S:” link in the WINE prefix and creating a symlink to their main SSD, which suggests that the issue is related to how Proton is interpreting the existing drive structure. Let's dig deeper into the potential causes of this behavior and explore why Proton might be choosing the “S:” drive over the usual “C:” drive.

The Symlink Connection

Interestingly, the user has several directories in their $HOME directory symlinked to locations on the HDD where the S: drive was mapped. These include:

  • Documents -> /run/media/merc/Data/Documents
  • Downloads -> /run/media/merc/Data/Downloads
  • Pictures -> /run/media/merc/Data/Pictures
  • Videos -> /run/media/merc/Data/Videos

This is a key piece of information! The presence of these symlinks might be influencing Proton's drive mapping logic. It's possible that Proton is detecting these symlinks and incorrectly identifying the target drive (the HDD) as the primary installation location. This behavior is a departure from previous Proton versions, where the “C:” drive was typically the default. The user notes that they've been using this symlink setup for a long time without issues, suggesting that the change in behavior is recent and likely related to updates in GE-Proton10-10 or underlying WINE components. Understanding how Proton handles symlinks and mounted drives is essential for troubleshooting this issue. It's possible that Proton is prioritizing mounted drives or specific file system structures when determining drive mappings. The user's workaround of deleting the S: link and creating a symlink to their SSD further supports the idea that symlinks are playing a significant role in this issue. We need to investigate how Proton interacts with symlinks and identify if there are any recent changes that could explain this new behavior. The fact that this issue has only surfaced in the past few weeks with recent GE-Proton10 releases suggests that a change in Proton's drive mapping logic is the most likely culprit.

Reproduction Steps: How to Recreate the Drive S Issue

To try and replicate this issue, here are the steps the user outlined:

  1. Install ESO from Steam.
  2. Set the compatibility runner to GE-Proton10-10.
  3. (Optional but potentially crucial) Create some symlinks in your $HOME directory to a secondary drive mounted somewhere in Linux (like the user's HDD at /run/media/username/Data).
  4. Run ESO for the first time and observe if it prompts for an installation directory on the S: drive.

It's important to note that the user suspects the symlinks are a key factor in triggering this behavior. While simply installing ESO and running it with GE-Proton10-10 might not be enough to reproduce the issue, the presence of symlinks to a secondary drive seems to be a crucial element. This is why step 3 is highlighted as potentially crucial. The act of creating these symlinks might be influencing Proton's drive mapping logic, causing it to incorrectly identify the secondary drive as a potential installation location. This underscores the importance of understanding how Proton interacts with the file system and how it determines drive mappings within its virtualized Windows environment. If you're experiencing this Drive S issue, trying to reproduce it with and without these symlinks can help isolate the problem and confirm whether they are indeed a contributing factor. Furthermore, attempting to reproduce the issue on different systems and with different Proton versions can provide valuable insights into the underlying cause. By systematically testing and replicating the issue, we can gather more information and develop a more effective solution.

Potential Causes and Troubleshooting Steps

So, what's going on here? Let's break down the potential causes and how we can troubleshoot this Drive S issue.

1. Recent Changes in Proton's Drive Mapping Logic

As the user pointed out, Proton's handling of virtual drives has evolved over time. It's possible that recent changes in GE-Proton10-10 have altered how Proton detects and maps drives, especially when symlinks and multiple drives are involved. To troubleshoot this, you could try:

  • Testing with older Proton versions: Try running ESO with older versions of Proton or GE-Proton to see if the issue persists. If it doesn't occur in older versions, it strongly suggests that a recent change is the culprit.
  • Checking Proton's release notes: Review the release notes for GE-Proton10-10 and subsequent versions to see if there are any mentions of changes related to drive mapping or file system handling. This can provide clues about potential causes.
  • Searching Proton's issue tracker: Look for existing bug reports or discussions about similar issues on Proton's GitHub issue tracker. Other users might have encountered the same problem and shared their findings or solutions. This is a valuable resource for troubleshooting Proton-related issues, as it allows you to leverage the collective knowledge of the Proton community.

2. Symlink Interference

The symlinks in the user's $HOME directory are a prime suspect. Proton might be following these links and incorrectly identifying the HDD as a valid installation target. To investigate this:

  • Temporarily removing the symlinks: Try temporarily removing the symlinks and see if the issue goes away. This is a simple but effective way to determine if the symlinks are indeed the cause. If removing the symlinks resolves the issue, you can be confident that they are playing a role.
  • Creating different symlinks: Experiment with creating symlinks to different locations or using different types of symlinks to see how Proton reacts. This can help you understand the specific conditions that trigger the issue.
  • Examining Proton's logs: Check Proton's logs for any messages related to drive mapping or symlink handling. This can provide valuable insights into how Proton is interpreting the file system structure. Proton logs often contain detailed information about the processes and decisions that Proton is making, which can be invaluable for troubleshooting complex issues like this one.

3. WINE Prefix Configuration

Proton uses WINE prefixes to create isolated Windows environments for each game. It's possible that the WINE prefix configuration is somehow contributing to the issue. To check this:

  • Creating a fresh WINE prefix: Try creating a fresh WINE prefix for ESO and see if the issue persists. This can help rule out any configuration issues within the existing prefix. A fresh WINE prefix will have default settings, which can help isolate the problem if it's related to a specific configuration setting.
  • Inspecting the WINE prefix: Examine the WINE prefix directory to see how the drives are mapped. You can use the winecfg command to access the WINE configuration tool and inspect the drive mappings. This will show you how WINE is currently mapping drives and directories within the virtual environment. Looking at the drive mappings can reveal if the S: drive is intentionally mapped to the HDD or if there's an unexpected configuration.

4. Mounted Drive Handling

How Proton handles mounted drives might also be a factor. To investigate this:

  • Trying different mount options: Experiment with different mount options for the HDD to see if it affects Proton's behavior. For example, you could try using different file system types or mount flags.
  • Mounting the drive in a different location: Try mounting the HDD in a different location in the file system and see if the issue persists. This can help determine if the mount point is influencing Proton's drive mapping logic. If mounting the drive in a different location resolves the issue, it suggests that the original mount point might be causing a conflict or misinterpretation.

Conclusion: Unraveling the Mystery of Drive S

The Drive S issue in Proton, particularly with GE-Proton10-10, is a curious problem that seems to be influenced by a combination of factors, including recent changes in Proton, symlinks, WINE prefix configurations, and how mounted drives are handled. By systematically troubleshooting using the steps outlined above, you can hopefully identify the root cause and get your games running smoothly. Remember to share your findings and any workarounds you discover with the Proton community, as this helps everyone benefit from collective knowledge. Keep experimenting, keep troubleshooting, and happy gaming, guys! This issue highlights the complexity of compatibility layers like Proton and the importance of understanding how different system components interact. While it can be frustrating to encounter such issues, the process of troubleshooting and finding solutions ultimately contributes to the ongoing improvement of Proton and the gaming experience on Linux.
