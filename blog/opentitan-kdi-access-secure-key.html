OpenTitan KDI Access: Secure Key Derivation & OTP Secrets

Aug 9, 2025 by ADMIN 58 views

Hey everyone! Today, we're diving deep into a crucial aspect of OpenTitan's security architecture: the Key Derivation Interface (KDI). This interface plays a vital role in deriving static and ephemeral scrambling keys for FLASH and SRAM, ensuring the confidentiality and integrity of your data. However, a key question arises: how can we, as software developers, access this powerful interface, especially when dealing with external SPI flash devices and the need for secure key storage?

Understanding the Key Derivation Interface (KDI)

The Key Derivation Interface (KDI), as highlighted in the otp_ctrl Theory of Operation documentation, acts as a central hub for key management within OpenTitan. It interacts with several critical components, including the scrambling datapath, the EDN (Entropy Distribution Network), and the secure partitions within OTP (One-Time Programmable) memory. These partitions hold the root keys essential for scrambling operations. The primary function of the KDI is to derive both static and ephemeral scrambling keys, which are then used to protect data stored in FLASH and SRAM. This process ensures that sensitive information remains confidential and resistant to unauthorized access.

The KDI's role extends beyond simply generating keys; it also ensures the secure and controlled derivation of these keys. By leveraging the EDN, the KDI incorporates randomness into the key generation process, making it significantly harder for attackers to predict or compromise the keys. Furthermore, the interaction with secure OTP partitions ensures that the root keys themselves are protected from unauthorized access or modification. This multi-layered approach to key management is a cornerstone of OpenTitan's security architecture, providing a robust defense against various attack vectors.

The importance of the KDI becomes even more apparent when considering scenarios involving external storage, such as SPI flash devices. In these cases, the KDI can be used to derive encryption keys that protect the data stored on the external device. This ensures that even if the external storage is compromised, the data remains secure. However, the challenge lies in accessing the KDI from software, which is where the current limitations and potential solutions come into play. Understanding the KDI's functionality and its place within the broader security architecture is crucial for developers looking to leverage its capabilities for secure key management.

The Challenge: Software Access to the KDI

The main challenge we're tackling today is the apparent lack of direct software access to the Key Derivation Interface (KDI). While the documentation clearly outlines the KDI's crucial role in key derivation, there's a noticeable absence of documented APIs or DIF (Device Interface Functions) calls that would allow software to interact with it directly. This raises a significant question: how can we leverage the KDI's capabilities, particularly when dealing with scenarios that require encrypting data on external devices like SPI flash?

Imagine a scenario where you have an external SPI flash device connected to your OpenTitan system via the SPI Host peripheral. You want to encrypt the data stored on this flash device using a key derived from OTP (One-Time Programmable) memory. This is a common security requirement, especially when dealing with sensitive data. The ideal solution would be to use the KDI to derive an encryption key from material stored in one of the OTP's secret partitions. These partitions are specifically designed to hold sensitive information like root keys, and their contents are scrambled to prevent unauthorized access. However, without a clear path to access the KDI from software, this approach seems to be blocked.

This limitation poses a significant hurdle for developers who need to implement robust security measures in their OpenTitan-based systems. Without direct access to the KDI, it becomes challenging to securely manage keys for external storage or other peripherals. This, in turn, can impact the overall security posture of the system. Therefore, exploring potential solutions and understanding the rationale behind the current limitations is crucial for advancing the security capabilities of OpenTitan. The lack of software access to the KDI highlights a critical gap in the current implementation, and addressing this gap is essential for unlocking the full potential of OpenTitan's security features.

Exploring Solutions and Future Directions for KDI Access

So, what are the possible solutions, guys? While direct software access to the Key Derivation Interface (KDI) might not be readily available right now, let's explore some potential avenues and future directions for addressing this challenge. We need to think creatively about how we can securely derive keys from OTP material for use with external devices and other peripherals.

One potential approach could involve implementing a dedicated hardware module or coprocessor that handles key derivation on behalf of the software. This module could act as an intermediary between the software and the KDI, providing a secure and controlled interface for key derivation operations. The software would send a request to the module, specifying the desired key type and the source OTP partition. The module would then interact with the KDI to derive the key and return it to the software in a secure manner. This approach would isolate the sensitive key derivation process from the main software execution environment, reducing the risk of compromise.

Another possibility is to develop a set of secure APIs or DIF calls that provide limited access to the KDI's key derivation functionality. These APIs could be carefully designed to prevent misuse and ensure that keys are only derived under specific, authorized conditions. For example, the APIs might require authentication or authorization before allowing key derivation, or they might limit the types of keys that can be derived. This approach would provide a more direct path for software to access the KDI, but it would also require careful consideration of the security implications.

Looking further into the future, it's possible that OpenTitan's architecture could evolve to include a more comprehensive key management framework. This framework could provide a standardized way to manage keys across the entire system, including both internal and external storage. It might include features like key rotation, key revocation, and secure key storage. Such a framework would greatly simplify the process of managing keys in OpenTitan and would make it easier to implement robust security measures. These are just a few ideas, and the actual solution may involve a combination of these approaches or other innovative techniques. The key is to continue exploring the possibilities and working towards a secure and flexible key management solution for OpenTitan.

Securely Storing and Deriving Keys from OTP: Addressing Concerns

Beyond accessing the Key Derivation Interface (KDI), another critical aspect of secure key management is the ability to store and derive keys from OTP (One-Time Programmable) memory itself. A key concern arises around the ownership and control of the secret partitions within OTP, particularly in the context of the Creator-Owner model employed by OpenTitan. Let's delve into this issue and explore potential solutions to ensure that Owners have the necessary control over their key material.

According to the Direct Access Memory Map table in the otp_ctrl Programmer's Guide, the secret partitions are typically written to by the time the chips are sold by the Creator to the Owner. This raises a significant question: how can Owners ensure that their key material is the only material stored in these partitions, and that the Creator does not retain any copies of the keys? Even if the Creator is willing to flash the Owner's key material in the factory, there are still concerns about the Creator's internal key handling processes and the potential for key retention.

To address this concern, a crucial requirement is the ability for Owners to flash their own key material into the secret OTP partitions, without relying on the Creator. This would provide Owners with full control over their keys and eliminate the risk of key compromise during the manufacturing process. One way to achieve this is for chip manufacturers to offer chips with the CREATOR_ROOT_KEY partition unprogrammed. This would allow Owners to program their own root keys into the OTP, ensuring that they are the sole holders of these critical secrets.

Another approach could involve implementing a secure key exchange mechanism between the Creator and the Owner. This mechanism would allow the Owner to securely provision their keys to the Creator for flashing, without revealing the actual key material. This could be achieved using cryptographic techniques such as key wrapping or secure multi-party computation. However, this approach would still require trust in the Creator's implementation of the key exchange mechanism.

Ultimately, the most secure solution is to provide Owners with the ability to program their own keys into the OTP, without any involvement from the Creator. This ensures the highest level of control and transparency over key management. By offering chips with unprogrammed secret partitions, manufacturers can empower Owners to take full ownership of their security and mitigate the risks associated with key handling during the manufacturing process. This is a critical step towards building truly secure systems with OpenTitan.

Questions About Secret Partitions: Ensuring Ownership and Control

Following up on the discussion about secret partitions in OTP, a key question remains: will Earlgrey chip producers sell chips with unprogrammed CREATOR_ROOT_KEY partitions? This is crucial for ensuring that Owners have full control over their key material and can confidently manage their security posture. Let's break down why this question is so important and what the implications are.

If the CREATOR_ROOT_KEY partition is pre-programmed by the Creator, it introduces a potential vulnerability. Even if the Creator flashes the Owner's key material later, the fact that the Creator had access to the OTP and potentially programmed their own root key raises concerns about key retention and potential backdoors. Owners need assurance that their keys are the only keys present in the secret partitions, and that no other entity has access to them.

Selling chips with unprogrammed CREATOR_ROOT_KEY partitions allows Owners to take complete ownership of their security. They can program their own root keys into the OTP, ensuring that these keys are never exposed to the Creator or any other third party. This provides a strong foundation for building secure systems, as it eliminates the risk of key compromise during the manufacturing process.

Furthermore, this approach aligns with the principles of transparency and accountability, which are central to OpenTitan's mission. By giving Owners control over their key material, OpenTitan empowers them to build trust in their systems and demonstrate their security posture to their customers and stakeholders. This is particularly important in security-sensitive applications where the integrity and confidentiality of data are paramount.

The availability of chips with unprogrammed CREATOR_ROOT_KEY partitions is not just a technical detail; it's a fundamental requirement for building truly secure systems with OpenTitan. It's a question of trust, control, and accountability. Therefore, it's crucial for chip producers to address this need and provide Owners with the tools they need to manage their security effectively. This will enable the widespread adoption of OpenTitan in a variety of applications, from embedded systems to cloud infrastructure.
