Unclaimed Entities Plugin: Simplify Backstage Adoption

Aug 2, 2025 by ADMIN 55 views

Hey guys! Ever felt like wrangling a massive tech ecosystem is like herding cats? Backstage, with its catalog-info.yaml flow, makes adopting it within teams relatively smooth. But what about those sprawling organizations where figuring out the scope of adoption is half the battle? That's where the Unclaimed Entities plugin comes in ‚Äì a potential game-changer for large enterprises aiming to embrace Backstage.

The Challenge: Scaling Backstage Adoption in Large Organizations

In sprawling organizations, gaining momentum for change requires a clear understanding of the problem's magnitude. Think about it: you're trying to convince various teams to adopt Backstage, but without a comprehensive view of existing entities, it's like navigating a maze blindfolded. This is particularly true when dealing with numerous repositories, APIs, servers, and other resources scattered across different systems. Often, the initial hurdle is simply identifying what's out there and whether it's properly integrated into the Backstage ecosystem.

That's where the concept of Unclaimed Entities becomes incredibly valuable. By providing a clear picture of resources not yet claimed by Backstage, you can effectively showcase the scope of the adoption challenge. This visibility helps in several ways:

  • Quantifying the Problem: Instead of vague estimates, you have concrete data on the number of unclaimed entities, which makes it easier to justify the effort required for Backstage adoption.
  • Prioritizing Efforts: Knowing which repositories or resources are unclaimed allows you to focus on the most critical areas first, ensuring maximum impact with your adoption initiatives.
  • Tracking Progress: As teams claim entities and integrate them into Backstage, you can track the reduction in unclaimed entities, providing a tangible measure of your success.
  • Identifying Gaps and Opportunities: Unclaimed entities might reveal areas where documentation is lacking, ownership is unclear, or resources are simply forgotten. Addressing these issues can lead to broader improvements in your organization's IT landscape.

In essence, the Unclaimed Entities concept transforms the adoption challenge from a nebulous goal into a concrete, measurable objective. It empowers organizations to make informed decisions, allocate resources effectively, and drive Backstage adoption with greater confidence. This leads us to the core idea behind the proposed plugin: leveraging existing catalog processors to identify these unclaimed resources.

The Solution: Introducing the Unclaimed Entities Plugin

The Unclaimed Entities plugin is designed to tackle this challenge head-on. The core idea, a big shoutout to @awanlin for this brilliant concept, is to create a catalog Provider that leverages existing catalog processors ‚Äì the same ones you're already using for things like Azure DevOps and GitHub. The magic happens when the plugin scans these systems: for any repositories that don't have a catalog-info.yaml file, it registers them as a new Catalog EntityType called 'Unclaimed'.

Think of it this way: the plugin acts like a diligent auditor, checking your existing systems for resources that haven't yet been brought into the Backstage fold. This provides a powerful way to visualize the extent of your Backstage adoption gap and prioritize your efforts. This approach offers several key advantages:

  • Leverages Existing Infrastructure: By utilizing existing catalog processors, the plugin minimizes the need for new configurations and integrations. This makes it easier to deploy and manage.
  • Provides a Unified View: The plugin consolidates information about unclaimed entities from various sources into a single view within Backstage. This eliminates the need for manual tracking and spreadsheets.
  • Supports Multiple Entity Types: While the initial focus is on repositories, the plugin can be extended to identify other unclaimed entities such as servers, APIs, and services.
  • Drives Adoption: By highlighting unclaimed entities, the plugin encourages teams to take ownership of their resources and integrate them into Backstage.

The Unclaimed Entities plugin addresses a critical need in large organizations by providing a clear and actionable view of resources not yet managed by Backstage. By automating the discovery process and integrating seamlessly with existing systems, the plugin simplifies the adoption process and empowers teams to take control of their digital landscape.

Diving Deeper: Unclaimed Entity Types and Metadata

One of the key strengths of the Unclaimed Entities approach is its flexibility. The Unclaimed entity type isn't limited to just repositories; it's designed to be adaptable to a wide range of resources. The Unclaimed entity type can use the common metadata fields, this allows them to be used for other unclaimed entities, such as servers, APIs, or anything else you'd like to configure. This adaptability is crucial because different organizations have different priorities and types of resources they need to manage. The plugin's architecture allows you to define and track unclaimed entities that are most relevant to your specific context.

For example, in addition to repositories, you might want to track unclaimed:

  • APIs: Identifying APIs that haven't been documented or integrated into your API catalog can be critical for ensuring consistency and discoverability.
  • Servers: Unclaimed servers might represent forgotten infrastructure or potential security risks. Tracking them can help you optimize resource utilization and improve security posture.
  • Databases: Similarly, unclaimed databases could indicate data silos or potential compliance issues. Bringing them into Backstage can enhance data governance and accessibility.
  • Services: Identifying services that aren't properly registered or managed can improve service reliability and maintainability.

The ability to use common metadata fields across different unclaimed entity types is also a significant advantage. This allows you to:

  • Establish a Consistent Taxonomy: Define a standard set of metadata fields that apply to all unclaimed entities, making it easier to search, filter, and analyze them.
  • Create Meaningful Dashboards: Build dashboards that provide insights into the overall state of unclaimed resources, regardless of their type.
  • Automate Workflows: Trigger automated actions based on metadata values, such as sending notifications or creating tickets.

The flexibility of the Unclaimed entity type and its support for common metadata fields make the plugin a powerful tool for managing a diverse range of resources within Backstage. By tailoring the plugin to your specific needs, you can gain a comprehensive view of your digital landscape and drive adoption across your organization.

Future Directions: Module Extension Points and Custom Providers/Processors

Looking ahead, the vision for the Unclaimed Entities plugin extends beyond its initial capabilities. A key area of exploration is the development of Module Extension points. We should investigate Module Extension points to allow people to add their own unclaimed providers/processors. This would empower users to extend the plugin's functionality by adding their own unclaimed providers/processors. This would open up a world of possibilities, allowing organizations to tailor the plugin to their unique environments and integrate it with a wider range of systems.

Imagine being able to:

  • Integrate with Custom Systems: Connect the plugin to proprietary systems or internal tools that aren't supported by default.
  • Develop Specialized Providers: Create providers that target specific types of resources, such as cloud resources, legacy systems, or IoT devices.
  • Implement Custom Processing Logic: Add custom logic to identify and process unclaimed entities based on your organization's specific requirements.

By providing Module Extension points, the plugin would become a truly extensible platform for managing unclaimed resources. This would foster a vibrant ecosystem of community contributions, with users sharing their custom providers and processors to address a wide range of use cases.

The initial focus of the plugin will be on scanning Azure DevOps, and providing the entities to the catalog. This will provide a solid foundation for future development and allow the community to build upon the core functionality. However, the long-term goal is to support a wide range of providers and processors, making the plugin a versatile tool for any organization adopting Backstage.

The exploration of Module Extension points is a critical step in realizing this vision. It would transform the plugin from a standalone tool into a powerful platform for managing unclaimed resources, empowering organizations to take control of their digital landscape and drive Backstage adoption across their entire ecosystem.

Getting Started: Azure DevOps Integration and Beyond

The initial implementation of the Unclaimed Entities plugin will focus on scanning Azure DevOps repositories. This is a logical starting point, as Azure DevOps is a widely used platform for software development and provides a rich set of APIs for accessing repository metadata. The initial plugin will just scan Azure Devops, and provide the entities to the catalog.

This initial integration will provide a valuable proof of concept and demonstrate the core functionality of the plugin. It will allow users to:

  • Automatically Discover Unclaimed Repositories: Identify repositories in Azure DevOps that don't have a catalog-info.yaml file.
  • Visualize Unclaimed Entities in Backstage: See a list of unclaimed repositories within the Backstage catalog, providing a clear view of the adoption gap.
  • Prioritize Adoption Efforts: Focus on integrating the most critical unclaimed repositories into Backstage first.

However, the long-term vision for the plugin extends far beyond Azure DevOps. As mentioned earlier, the goal is to support a wide range of providers and processors, allowing users to identify unclaimed entities across their entire organization. This includes:

  • GitHub: Integrating with GitHub, another popular platform for software development, is a natural next step.
  • Cloud Providers: Supporting cloud platforms like AWS, Azure, and GCP would allow users to identify unclaimed cloud resources, such as virtual machines, databases, and storage buckets.
  • API Gateways: Integrating with API gateways would enable the discovery of unclaimed APIs.
  • Custom Systems: The Module Extension points, discussed earlier, would allow users to integrate with proprietary systems and internal tools.

By starting with Azure DevOps and gradually expanding to other providers, the Unclaimed Entities plugin will become an increasingly valuable tool for organizations adopting Backstage. The initial Azure DevOps integration will provide a solid foundation for future development, paving the way for a comprehensive solution for managing unclaimed resources across the entire digital landscape.
