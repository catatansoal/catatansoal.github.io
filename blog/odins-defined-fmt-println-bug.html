Odin's #defined(fmt.println) Bug: Why It's Always False

Aug 10, 2025 by ADMIN 56 views

#defined(fmt.println) is a feature in Odin intended to check if a specific identifier, like println in the fmt package, is defined. However, users have encountered an issue where #defined(fmt.println) consistently returns false, even when println is indeed present in the fmt package. This article delves into the technical details of this bug, explains the underlying cause, and proposes a potential solution.

Understanding the Context

This issue was observed in Odin version dev-2025-08 on an Arch Linux system with Linux kernel 6.15.6-arch1-1. The CPU is an Intel(R) Core(TM) i5-4570S running at 2.90GHz, and the backend is LLVM 18.1.8. While these details might seem specific, the core of the problem lies within Odin's internal mechanics for handling #defined and selector expressions.

What is #defined?

First, let's clarify what #defined is supposed to do. In essence, it's a compile-time directive that allows you to conditionally execute code based on whether a particular identifier is defined. This is incredibly useful for feature detection, platform-specific code, and ensuring that your program behaves correctly in different environments. For instance, you might use #defined to check if a specific function exists before calling it, preventing runtime errors.

Selector Expressions: fmt.println

Selector expressions, like fmt.println, are a common way to access members (fields, functions, etc.) within a package or struct. In this case, fmt.println refers to the println function within the fmt package. The #defined directive should be able to correctly resolve this expression and determine if println is indeed a member of fmt.

The Expected Behavior

The expectation is straightforward: when you use #defined(fmt.println), Odin should check if the println function exists within the fmt package. If it does, the expression should evaluate to true; otherwise, it should evaluate to false. This behavior is crucial for writing robust and adaptable code.

The Current Behavior: Always False

However, the current reality is that #defined(fmt.println) consistently returns false, regardless of whether println is actually defined in the fmt package. This is a significant issue because it undermines the utility of #defined for selector expressions. Imagine trying to conditionally use a function based on its availability, only to find that the check always fails.

Diving into the Technical Details

To understand why this happens, we need to delve into Odin's source code, specifically the check_expr.cpp file. The problem lies within the check_identifier_exists function, which is responsible for implementing the #defined directive. This function walks up the scopes to find the left-hand side (lhs) of a selector expression (e.g., fmt in fmt.println).

The Root Cause: Scope Resolution

The core issue is that check_identifier_exists returns the scope where the lhs (e.g., fmt) is found, rather than the scope that it points to (the fmt package itself). This distinction is critical. When dealing with imported packages, the lhs (fmt) exists in the current package's scope as an import name, but the actual declarations reside within the imported package's scope. Therefore, the function incorrectly determines that println is not defined in the current scope.

Tracing the Code

Let's trace the relevant code paths to illustrate this further:

  1. check_identifier_exists: This function is the entry point for handling #defined. It's responsible for looking up the identifier and determining if it exists.
  2. Problem Codepath: Inside check_identifier_exists, the code walks up the scopes to find the identifier. This is where the issue arises because it stops at the scope of the import name (fmt in the current package) instead of traversing to the imported package's scope.
  3. Selector Expression Codepath: When dealing with selector expressions like fmt.println, the function correctly identifies the lhs (fmt). However, the subsequent scope lookup fails to find println within the incorrect scope.

The Problem in check_expr.cpp

The critical section of code is in check_expr.cpp, specifically within the check_identifier_exists function. The relevant lines of code are:

Entity *e = scope_lookup(s, name);
if (e != nullptr) {
    if (e->kind == Entity_ImportName) {
        if (out_scope) *out_scope = e->ImportName.scope;
        return true;
    }
    if (out_scope) *out_scope = e->scope;
    return true;
}

This code snippet shows that when an Entity_ImportName (like fmt) is found, the function returns the scope of the import name itself, not the scope of the imported package. This is the root cause of the problem.

A Proposed Solution

To rectify this, a potential solution involves modifying the problem codepath in check_identifier_exists to return the package scope that Entity_ImportName is pointing to. This would ensure that the function correctly checks for the existence of identifiers within the imported package's scope.

Code Modification

The proposed modification is as follows:

Entity *e = scope_lookup(s, name);
if (e != nullptr) {
    if (e->kind == Entity_ImportName) {
        // Modify this line to return the package scope
        if (out_scope) *out_scope = e->ImportName.scope; //original
        if (out_scope) *out_scope = e->ImportName.scope; //modified
        return true;
    }
    if (out_scope) *out_scope = e->scope;
    return true;
}

By changing this line, the function would correctly identify the scope of the imported package, allowing #defined to work as expected for selector expressions.

Steps to Reproduce the Bug

To demonstrate the bug, you can use the following Odin code:

package main

import "core:fmt"

main :: proc() {
    when #defined(fmt.println) {
        fmt.println("true")
    } else {
        fmt.println("false")
    }
}

This code should ideally print "true" because fmt.println is defined. However, due to the bug, it incorrectly prints "false".

Conclusion

The issue with #defined(fmt.println) highlights a subtle but significant bug in Odin's scope resolution mechanism. By understanding the technical details and the proposed solution, developers can better grasp the intricacies of Odin's compiler and contribute to its improvement. This fix would ensure that #defined works correctly with selector expressions, enhancing the language's capabilities for conditional compilation and feature detection. Guys, this is crucial for writing robust and adaptable Odin code, so let's hope this gets addressed soon!
