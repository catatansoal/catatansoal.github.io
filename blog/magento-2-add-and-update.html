Magento 2: Add And Update Created At/Updated At Columns

Aug 12, 2025 by ADMIN 56 views

Hey guys! Ever wondered how to automatically track when a record was created or last updated in your Magento 2 custom tables? Well, you've come to the right place! In this comprehensive guide, we'll walk you through the process of adding created_at and updated_at columns to your custom tables in Magento 2, and more importantly, how to automatically update these columns whenever you save data. This is super useful for auditing, reporting, and generally keeping track of your data's lifecycle. So, let's dive in!

Why Track Creation and Update Times?

Before we get our hands dirty with the code, let's quickly chat about why tracking created_at and updated_at is a really good idea. Think about it: you're building a cool extension for Magento 2, and you have custom tables storing all sorts of data. Knowing when a record was created and when it was last modified can be a game-changer.

  • Auditing: Need to know when a particular entry was created or modified? These timestamps give you a clear audit trail.
  • Reporting: Want to generate reports based on time periods? created_at and updated_at are your best friends.
  • Data Management: These timestamps help in identifying stale data, implementing data retention policies, and optimizing your database.
  • Debugging: When things go wrong (and they sometimes do!), these timestamps can help you trace back changes and pinpoint issues.

So, yeah, tracking these timestamps is a big win for data integrity and overall application health. It's like having a time machine for your data! Now that we're on the same page about the "why," let's jump into the "how."

Step 1: Modifying Your Database Schema

The first step in our journey is to modify your custom table's schema. We'll be adding two new columns: created_at and updated_at. These columns will store the timestamps of when a record was created and last updated, respectively. We will achieve this by creating an upgrade schema script. These scripts are Magento's way of managing database changes in a controlled and versioned manner.

Let's break down the process into smaller, manageable steps:

1. Create the UpgradeSchema Script

First, navigate to your module's Setup directory. If you don't have one, create it. Within the Setup directory, you'll create a PHP file named UpgradeSchema.php. The name is important, guys, so make sure you get it right! This file will contain the logic to modify your database schema. Here's a basic structure of the file:

<?php

namespace Vendor\Module\Setup;

use Magento\Framework\Setup\UpgradeSchemaInterface;
use Magento\Framework\Setup\ModuleContextInterface;
use Magento\Framework\Setup\SchemaSetupInterface;
use Magento\Framework\DB\Ddl\Table;

class UpgradeSchema implements UpgradeSchemaInterface
{
    public function upgrade(SchemaSetupInterface $setup, ModuleContextInterface $context)
    {
        $setup->startSetup();

        if (version_compare($context->getVersion(), '1.0.1', '<')) {
            // Your code to add the columns will go here
        }

        $setup->endSetup();
    }
}

Let's dissect this code:

  • namespace Vendor\Module\Setup;: This defines the namespace for your class. Replace Vendor and Module with your actual vendor and module names. Remember, namespaces are crucial in PHP for avoiding naming conflicts.
  • use ...;: These lines import the necessary classes and interfaces from the Magento framework. We need these to interact with the database schema.
  • class UpgradeSchema implements UpgradeSchemaInterface: This declares our class, which implements the UpgradeSchemaInterface. This interface requires us to define the upgrade method.
  • public function upgrade(SchemaSetupInterface $setup, ModuleContextInterface $context): This is the main method where we'll write our database modification logic. It takes two arguments: $setup (an instance of SchemaSetupInterface, which provides methods for schema manipulation) and $context (an instance of ModuleContextInterface, which provides information about the module's current version).
  • $setup->startSetup(); and $setup->endSetup();: These methods are important for wrapping your database modifications. They ensure that the database connection is properly managed and that the schema changes are applied atomically. Think of them as the opening and closing brackets for your database transaction.
  • if (version_compare($context->getVersion(), '1.0.1', '<')) { ... }: This is a crucial part of the upgrade script. It checks the current version of your module against a target version (1.0.1 in this example). This ensures that the schema changes are only applied if the module is older than the target version. This is how Magento handles schema upgrades in a versioned manner. Make sure to use an appropriate version number here!

2. Add the Columns

Now comes the fun part: adding the created_at and updated_at columns to your table. Inside the if block in your UpgradeSchema.php file, you'll add the following code:

            $tableName = $setup->getTable('your_custom_table'); // Replace with your table name

            if ($setup->getConnection()->isTableExists($tableName) == true) {
                $setup->getConnection()->addColumn(
                    $tableName,
                    'created_at',
                    [
                        'type' => Table::TYPE_TIMESTAMP,
                        'nullable' => false,
                        'default' => Table::TIMESTAMP_INIT,
                        'comment' => 'Created At'
                    ]
                );

                $setup->getConnection()->addColumn(
                    $tableName,
                    'updated_at',
                    [
                        'type' => Table::TYPE_TIMESTAMP,
                        'nullable' => false,
                        'default' => Table::TIMESTAMP_INIT_UPDATE,
                        'comment' => 'Updated At'
                    ]
                );
            }

Let's break this down piece by piece:

  • $tableName = $setup->getTable('your_custom_table');: This line retrieves the full table name using the $setup->getTable() method. You must replace 'your_custom_table' with the actual name of your custom table in Magento's database. This method ensures that you're using the correct table name, even if Magento's table name prefix is in use.
  • if ($setup->getConnection()->isTableExists($tableName) == true) { ... }: This is a safety check to ensure that the table actually exists before we try to add columns to it. This is a good practice to prevent errors if the module is installed on a database that doesn't have the table.
  • $setup->getConnection()->addColumn(...): This is the core method for adding a column to the table. It takes three arguments:
    • $tableName: The name of the table to modify.
    • 'created_at': The name of the column to add.
    • [...]: An array of column options. Let's look at these options in more detail:
      • 'type' => Table::TYPE_TIMESTAMP: Specifies the data type of the column as a timestamp. This is the correct data type for storing date and time values.
      • 'nullable' => false: Specifies that the column cannot contain NULL values. This is a good practice for created_at and updated_at columns, as we always want them to have a value.
      • 'default' => Table::TIMESTAMP_INIT: This is the magic that automatically sets the created_at column to the current timestamp when a new row is inserted. Table::TIMESTAMP_INIT is a constant that tells Magento to use the current timestamp as the default value.
      • 'comment' => 'Created At': Adds a comment to the column. This is useful for documenting your database schema.
  • The code for adding the updated_at column is very similar, with a few key differences:
    • 'default' => Table::TIMESTAMP_INIT_UPDATE: This is the real magic! This tells Magento to automatically update the updated_at column to the current timestamp whenever the row is updated. This is exactly what we want for our updated_at column.

3. Update module.xml

Magento uses module.xml to keep track of module versions. You'll need to update the <module> tag in your module's etc/module.xml file to reflect the new version number. This tells Magento that your module has been updated and that the upgrade scripts need to be run.

Open your etc/module.xml file (create it if it doesn't exist) and update the setup_version attribute:

<?xml version="1.0"?>
<config xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance" xsi:noNamespaceSchemaLocation="urn:magento:framework:Module/etc/module.xsd">
    <module name="Vendor_Module" setup_version="1.0.1">
    </module>
</config>
