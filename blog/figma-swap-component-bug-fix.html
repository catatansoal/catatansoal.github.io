Understanding the Figma Swap Component Bug

At its core, this Figma swap component bug manifests as an unintended border stroke appearing on elements after they've been swapped using the .swap component functionality. This issue isn't consistent across the board, making it a bit tricky to pin down. However, it's been observed specifically within components like Details, Dialog, and Card. When designers attempt to add or replace elements using the swap component feature, an unwanted border suddenly pops up, disrupting the visual harmony and intended design.

This unexpected behavior stems from an overwrite state during the component swapping process. Basically, it seems like the border settings from a previous state or component are being inadvertently applied to the newly swapped element. This can be particularly frustrating because it forces designers to manually reset the border instance on every occurrence of the swapped component, adding extra steps and potential for human error. We need to address this to ensure a smoother, more predictable design workflow for everyone using these components.

The Root Cause: Overwrite State on Swap Components

The root of the problem lies in how Figma handles the inheritance and overriding of properties during component swaps. It seems that in certain scenarios, the border properties are not being properly reset, resulting in this unwanted carry-over effect.
