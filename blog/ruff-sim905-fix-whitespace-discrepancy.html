Understanding the SIM905 Issue

At the heart of the matter is the SIM905 rule in Ruff, which deals with splitting static strings. The problem arises because Python's str.split and str.rsplit methods treat the Unicode characters U+001C through U+001F as whitespace. However, the split-static-string implementation in Ruff, identified as SIM905, doesn't recognize these characters as whitespace. This discrepancy can lead to unexpected changes in program behavior after Ruff applies its fixes. It's like thinking you're cleaning up your room, but your cleaning robot has a different definition of what's messy!

To understand this better, let's break down the technical details. SIM905 leverages Rust's str::split_whitespace, str::trim_start, and str::trim_end methods. These Rust methods define whitespace according to the Unicode [:White_Space:] property. On the other hand, Python's definition of whitespace is broader, encompassing [[:bc=B:][:bc=S:][:bc=WS:][:Zs:]]. This broader definition includes those pesky U+001C to U+001F characters, which are known as Information Separators. So, while Python sees them as valid split points, Ruff's underlying Rust methods do not. This difference in interpretation is where our trouble begins.

Illustrating the Problem with an Example

Let's look at a real-world example to see the impact. Consider the following Python code:

print("S\x1cP\x1dL\x1eI\x1fT".split())

Diving Deeper into Whitespace Definitions

The core of this issue lies in the differing definitions of whitespace between Python and the Rust methods used by Ruff's SIM905 rule. Python defines whitespace more broadly, including characters from the categories [:bc=B:], [:bc=S:], [:bc=WS:], and [:Zs:]. These categories encompass a variety of whitespace characters, including the U+001C to U+001F range, which are control characters used as information separators.

In contrast, the Rust methods str::split_whitespace, str::trim_start, and str::trim_end adhere to the Unicode [:White_Space:] property. This property includes common whitespace characters like spaces, tabs, and newlines, but it excludes the U+001C to U+001F characters. This narrower definition is the root cause of the discrepancy we're seeing. It's like having two dictionaries, one with a comprehensive list of words and another with a more selective one. When you use the selective dictionary to fix the comprehensive one, you might end up losing some valid words!

This difference might seem subtle, but it has significant implications for code correctness. Imagine a scenario where you're processing data that uses these information separator characters.
