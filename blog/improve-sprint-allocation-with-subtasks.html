Improve Sprint Allocation With Subtasks And Git History

Jul 16, 2025 by ADMIN 56 views

This article discusses enhancing our sprint allocation logic to accurately reflect contributions from multiple team members, especially when they work on subtasks or contribute code directly via Git. The current system primarily focuses on parent stories, which can overlook valuable effort data from subtasks and direct code commits. This leads to manual effort in tracking contributions and can result in inaccuracies in time allocation. We aim to improve this process by automatically incorporating subtask participants and Git history into the allocation calculations, ensuring a more comprehensive and fair distribution of effort.

The Challenge with Current Sprint Allocation

Currently, our sprint allocation logic focuses solely on parent stories, excluding Jira subtasks from the final time allocation. This approach, while simplifying calculations, inadvertently hides crucial effort data when multiple team members contribute through subtasks linked to a single parent story. The allocation percentage is calculated only for the parent story, but we find ourselves manually performing several steps to account for all contributors. These steps include checking which team members worked on related subtasks, adding up their time or story points, including them in the time distribution, and even manually inspecting the Git history to identify code commits related to the main story when subtasks are absent. This manual process is not only inefficient but also prone to errors, highlighting the need for a more automated and accurate solution.

The Inefficiency of Manual Tracking

The current manual process is time-consuming and resource-intensive. It requires a significant amount of effort to track individual contributions, especially in projects with numerous subtasks or direct code commits. This manual effort detracts from the team's productivity and can lead to delays in project completion. Furthermore, the manual nature of the process increases the risk of errors, potentially leading to inaccurate time allocations and an unfair distribution of credit for the work done. Therefore, automating this process is crucial for improving efficiency and ensuring fairness in sprint allocation.

The Risk of Overlooking Subtask Contributions

By excluding subtasks from the allocation calculation, we risk overlooking significant contributions made by team members. Subtasks often represent a substantial portion of the overall effort required for a story, and failing to account for this effort can lead to an underestimation of the work involved and an inaccurate reflection of individual contributions. This can have a negative impact on team morale and motivation, as team members may feel that their efforts are not being properly recognized. Therefore, it is essential to incorporate subtask contributions into the allocation process to ensure a fair and accurate representation of the work done.

The Difficulty in Identifying Git Contributions

When no subtasks exist, the current process relies on manually inspecting the Git history to identify code commits related to the main story. This is a cumbersome and time-consuming task, especially for projects with a large number of commits. It requires a deep understanding of the codebase and the commit history, and it is prone to errors. Furthermore, it can be challenging to map Git commits to specific team members, especially if commit messages are not consistently formatted or if multiple team members have contributed to the same commit. Therefore, automating the identification of Git contributions is crucial for improving efficiency and accuracy in sprint allocation.

Objective: Enhancing Classification and Allocation Logic

Our primary objective is to enhance the classification and allocation logic to accurately capture contributions from all team members, regardless of whether they worked on subtasks or committed code directly. We aim to achieve this by rolling up subtask participants and effort into the parent story, counting all engineers who worked on subtasks as contributors to the main story, and checking Git history for commits related to the story when no subtasks exist. The final TimeAllocation output should include all contributors with proportional splits, and the updated CSV and Jira push should reflect this richer participant breakdown. This enhancement will ensure a more comprehensive and fair distribution of effort, leading to improved team morale and productivity.

Rolling Up Subtask Participants and Effort

The first key enhancement is to roll up subtask participants and effort into the parent story. This means that the system will automatically detect linked subtasks for each parent story and gather all assignees from those subtasks. These assignees will then be included in the total TimeAllocation for that parent story. This will ensure that all team members who contributed to the story, whether through subtasks or direct work on the parent story, are properly recognized and accounted for in the allocation calculations. This is a crucial step in ensuring a fair and accurate distribution of effort.

Incorporating Git History for Direct Commits

To capture contributions from team members who commit code directly without creating subtasks, the system will check Git history for commits referencing the story. This involves looking up the Git history for commits referencing the story key (e.g., FN-123). The system will then resolve contributor Git usernames and map them to known users in the system, adding them to the allocation. This enhancement will ensure that all code contributions are properly accounted for, regardless of whether they are associated with a subtask. This is particularly important for projects with a large number of direct code commits, as it ensures that all team members who contributed code are recognized for their efforts.

Proportional Time Allocation

Once all contributors have been identified, the total time percentage will be split proportionally among them. This ensures a fair and equitable distribution of effort, reflecting the contributions of each team member. Two options are being considered for determining the proportional split: an even split by default (e.g., 2 people â†’ 50% each) and weighting by subtask points if available. The final TimeAllocation output will list all contributors per story, ensuring transparency and accountability. This enhancement will ensure that all team members are recognized for their contributions and that the allocation of effort is fair and equitable.

Automatic Detection of Subtasks and Git Contributions

One of the key acceptance criteria is the automatic detection of linked subtasks for each parent story. This means that the allocation logic must be able to identify all subtasks associated with a parent story and gather the assignees from those subtasks. Similarly, the system must be able to automatically look up Git history for commits referencing the story key if no subtasks exist. This automatic detection is crucial for streamlining the allocation process and ensuring that all contributions are accurately captured. Without this automation, the process would remain manual and prone to errors, defeating the purpose of the enhancement.

Proportional Time Allocation and Output

Another critical acceptance criterion is the proportional splitting of time among contributors. The total time percentage must be split proportionally among contributors, ensuring a fair and equitable distribution of effort. The final TimeAllocation output must list all contributors per story and sum to 100% when combined. This transparency is essential for building trust and ensuring that all team members are recognized for their contributions. Furthermore, the updated CSV must include a column for each contributor with their share, and the Jira push must log the time against each participant properly (if supported). This ensures that the allocation information is accurately reflected in all relevant systems.
