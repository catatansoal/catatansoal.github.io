Feature Request: Asset Editor for FHIR Questionnaires

Overview

Alright, so we're talking about adding an Asset Editor for FHIR Questionnaires to the DAK Dashboard within the @litlfred/sgex project. This falls under the category of "Other DAK Components." The goal? To create a user-friendly, web-based interface for managing and editing FHIR Questionnaire assets, which are located in input/questionnaires/*.json. Think of it as a one-stop-shop for handling all your questionnaire needs right within the dashboard.

Background

To really understand why this is important, let's break down the background.

  • First off, questionnaire assets follow the FHIR Questionnaire resource specification. That means they're structured in a specific way, adhering to FHIR standards, and are stored as JSON files.
  • There's a real need for a user-friendly, multilingual (i18n) editor. We need to be able to easily manage, edit, and validate these assets directly within the DAK Dashboard. No more jumping through hoops or using clunky tools! This is about streamlining the process and making it as intuitive as possible.
  • We've explored a bunch of FHIR Questionnaire editors, and after careful consideration, we landed on the LHC-Forms (LHC-Questionnaire) library. Why? Because it's open-source, actively maintained, React-compatible, and i18n-ready. Basically, it checks all the boxes for what we need.

Requirements

Let's get into the nitty-gritty of what this feature needs to do. We'll break it down into functional and non-functional requirements.

Functional Requirements

These are the core tasks the Asset Editor needs to handle. Think of these as the "must-haves" to make this feature a success.

  1. Questionnaire Asset Management

    • The editor should list all FHIR Questionnaire assets found under input/questionnaires/*.json. This is your inventory, so you need to see everything at a glance.
    • Users need to be able to create, edit, duplicate, and delete Questionnaire assets. Full control over your data is key.
    • Basic metadata display is essential. We're talking about things like name, id, title, status, and date last modified. This helps you keep track of your assets and quickly identify what you're looking for.

  2. Visual Questionnaire Editing

    • We're integrating the LHC-Forms visual editor for FHIR Questionnaires. This is where the magic happens, giving you a visual way to interact with and modify your questionnaires.
    • Support for loading and saving Questionnaire assets directly from/to JSON files in input/questionnaires/ is crucial. This ensures seamless integration with our existing file structure.
    • The editor needs to support validation according to FHIR Questionnaire schema. This ensures that your questionnaires are compliant and error-free. This step is critical for ensuring data integrity and interoperability.

  3. Internationalization (i18n)

    • The interface must support multiple languages for both the UI and Questionnaire content. We're building this for a global audience, so language support is non-negotiable.
    • Leveraging LHC-Forms’ support for FHIR’s Questionnaire.language and item-level translation extensions is a smart move. This simplifies the process of handling translations within the FHIR framework.
    • We need a UI mechanism (e.g., dropdown) for users to select their preferred language. A simple and intuitive way to switch languages is a must-have.
    • When creating or editing Questionnaires, users must be able to add/edit translations for all text fields. This ensures that all content is available in multiple languages, maintaining consistency across different versions.

  4. Integration with DAK Dashboard

    • The asset editor must be accessible from the “Other DAK Components” section of the dashboard. This keeps everything organized and easy to find.
    • We'll use existing DAK Dashboard theming and authentication/authorization mechanisms. This ensures a consistent look and feel, as well as maintaining security.

  5. Usability

    • The editor must function well on modern browsers and be responsive. It should work seamlessly no matter what device or browser you're using.
    • We need to provide clear error messages and guidance when loading or saving fails, or when validation errors occur. Users need to know what went wrong and how to fix it.

  6. Extensibility

    • The solution should allow for future support of additional FHIR resource types as assets. We're building for the future, so we need to make sure the system can grow with our needs.

Non-Functional Requirements

These are the qualities that define how well the system performs. Think of them as the “nice-to-haves” that make the feature truly great.

  • Performance: The editor should load and save Questionnaire assets quickly, even for large forms. Nobody wants to wait around for ages while a questionnaire loads.
  • Security: Only authorized users may edit or manage assets. Security is paramount, so we need to ensure that only the right people have access.
  • Accessibility: We'll follow WCAG 2.1 AA guidelines where practical. This ensures that the editor is accessible to everyone, regardless of their abilities.
  • Open Source Compliance: All dependencies (including LHC-Forms) must be compatible with project licensing. We need to play nice with open-source licenses.

Technology Considerations

Let's talk tech. Here’s what we’re thinking:

  • We'll use LHC-Forms as the embedded editor component. It supports React embedding, FHIR Questionnaire JSON, and i18n via FHIR translation extensions – a perfect fit!
  • We'll store and manage assets as FHIR-compliant JSON files in input/questionnaires/. This keeps everything standardized and easy to work with.
  • We'll use SUSHI for any required FSH-to-JSON conversion, but this isn't required for a JSON-only workflow.

Open Questions

There are still a few questions we need to answer:

  • Should editing be in-place (directly in the repo) or require a draft/approval workflow? This impacts how changes are managed and reviewed.
  • Will multi-user collaboration or versioning be required in the MVP? Collaboration features can add complexity, but also value.
  • Should the editor support QuestionnaireResponse assets in addition to Questionnaires? Expanding support to responses could be a future enhancement.

Navigating the Open Questions in FHIR Questionnaire Editor Development

As we progress with the FHIR Questionnaire editor for the DAK Dashboard, it’s essential to address several open questions that could significantly influence the editor's functionality and workflow. These questions revolve around editing workflows, collaboration features, and the scope of asset support. Let’s explore these considerations in detail to ensure the development process aligns with user needs and project goals.

Editing Workflow: In-Place vs. Draft/Approval

One of the primary decisions is whether editing should occur in-place or follow a draft/approval workflow. In-place editing allows users to make changes directly to the repository, offering immediate updates. However, this approach lacks a formal review process, which can be problematic in collaborative environments.

A draft/approval workflow, on the other hand, introduces a review stage before changes are implemented. This ensures that all modifications are vetted, maintaining the integrity and accuracy of questionnaires. While this method adds a layer of security and oversight, it also introduces complexity and may slow down the editing process. The choice between these workflows depends on the organizational needs and the level of control required over questionnaire content.

For organizations prioritizing speed and agility, in-place editing may be suitable. However, for those in regulated industries or those requiring strict quality control, a draft/approval workflow may be necessary. A hybrid approach, offering both options based on user roles or questionnaire sensitivity, could also be considered.

Collaboration and Versioning: Essential for Teamwork

Another critical consideration is whether multi-user collaboration or versioning will be required in the Minimum Viable Product (MVP). Collaboration features allow multiple users to work on the same questionnaire simultaneously, fostering teamwork and efficiency. Versioning, on the other hand, provides a history of changes, allowing users to revert to previous versions if needed.

While collaboration and versioning can enhance productivity and reduce errors, they also introduce complexity to the development process. Implementing these features requires careful planning and robust technical solutions. For the MVP, it may be prudent to focus on core editing functionalities and defer collaboration and versioning to future iterations.

However, if collaborative editing is a key requirement for the target user base, it should be prioritized. Similarly, if the organization has strict version control requirements, implementing versioning from the outset may be necessary. A thorough understanding of user needs and organizational policies is essential for making informed decisions about these features.

Scope of Asset Support: Questionnaires and Beyond

The initial focus of the FHIR Questionnaire editor is on supporting Questionnaire assets. However, it’s important to consider whether the editor should also support QuestionnaireResponse assets. QuestionnaireResponses contain the answers provided by individuals completing a questionnaire. Supporting these assets could provide valuable insights and streamline data analysis.

Including QuestionnaireResponse support would require additional development effort and may introduce new complexities. For example, the editor would need to handle potentially sensitive patient data, necessitating robust security measures. Additionally, the user interface would need to accommodate the display and editing of response data.

For the MVP, it may be pragmatic to focus solely on Questionnaire assets and evaluate the feasibility of adding QuestionnaireResponse support in a later phase. However, if there is a clear need for managing responses within the dashboard, this feature should be considered for inclusion in the initial release.
